- do not make comments that just say to add `data-testid` properties
- do not make comments that are not actionable (e.g. something is well structured, or well documented, or the change was required due to a change in that same PR elsewhere). UNLESS the comment is really important
- do not suggest changing `string | number` or `string | number | null` types to plain `number` or `number | null`. This is an intentional pattern throughout the codebase: Prisma `Decimal` fields serialize as strings in JSON API responses, so these mixed types are required. All call sites already wrap with `Number()` where arithmetic is needed.
- do not suggest moving display-formatting logic (such as `pruneHotelName`) inside shared components like `BookingCard`. These components are reused in multiple contexts (e.g. dashboard and bookings list) where different formatting behavior is needed. Applying transformations at the call site is intentional.
- do not comment on GraphQL query strings being long or hard to read. These are single-line strings sent verbatim over HTTP and cannot be reformatted without breaking the request.
- do not suggest replacing hardcoded app-specific constants (e.g. the GitHub repo slug in the health dashboard) with environment variables. These values are fixed for this specific application and extracting them to env vars adds complexity without benefit.
- do not comment on or suggest changes to files in `prisma/migrations/`. These SQL files are auto-generated by `prisma migrate dev` and are never edited by hand.
- do not suggest using `prisma db push` instead of `prisma migrate`. The project uses the versioned migration workflow (`prisma migrate dev` / `prisma migrate deploy`) intentionally for production safety.
- do not suggest splitting large settings components (e.g. `credit-card-accordion-item.tsx`, `card-benefits-section.tsx`) into multiple files solely because they exceed a line-count threshold. These files contain tightly coupled sub-components and dialogs that belong together; splitting them adds indirection without improving clarity.
- when reviewing logging or observability code, be mindful of PII. Do not suggest adding user email addresses, names, IP addresses, or other personally identifiable information to log events or metrics. `userId` (an opaque internal ID) is acceptable; email, name, `from`/`to` addresses, and similar fields are not.
